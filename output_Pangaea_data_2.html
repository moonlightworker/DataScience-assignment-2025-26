2. Combine datasets to one table

In order to merge all data records into one large table, I limited the input to the entries that each data record should contain:

  • id
  • date
  • depth
  • location (longitude/latitude)
  • the chosen parameter (salinity, pH, temperature, oxygen, chlorophyll…)

2.1 Safe merging

In an attempt to address the various challenges encountered while working with the PANGAEA datasets, efforts have been made to fix every issue that arose. However, there are probably further issues yet to be encountered. Most of the issues are in regard of column names or data types.
Among other things, the following code adresses the problem when several columns are named after the parameter (e.g. oxygen concentration [µmol/l] and oxygen saturation [%]). To get around this, the parameter ID is compared with the corresponding unit (using the PANGAEA parameter list) and columns with the correct unit are transferred. If there are still several matching columns, the data record is skipped and a message is displayed (e.g. for data records with oxygen surface [µmol/l] and oxygen near the bottom [µmol/l]). In order to be able to continue using these data records, manual inspection and decision-making would be necessary.

# Specifying the column names
clean_names <- c(
  "event", "date", "depth",
  "longitude", "latitude",
  parameter
)

# Desired data types
column_types <- list(
  event     = as.character,
  date      = as.character,
  depth     = as.numeric,
  longitude = as.numeric,
  latitude  = as.numeric
)

# Add parameter column type
column_types[[parameter]] <- as.numeric

# Track skipped datasets
skipped <- 0

# Read, clean, rename, type-convert each file
combined_data <- file_list %>%
  lapply(function(file) { 
    
    read.delim(file) %>%
      mutate(
        id = str_extract(file, "(?<=dataset_)\\d+") %||% "unknown"
      ) %>%
      {
        df <- .

        # Rename messy columns
        for (clean in clean_names) {
          matched <- grep(clean, names(df),
                          ignore.case = TRUE, value = TRUE)

          if (length(matched) >= 1) {

            # If multiple parameter columns match...
            if (clean == parameter && length(matched) > 1) {

              # search for parameter
              cand <- matched[grepl(clean, matched, ignore.case = TRUE)]
              
              # prioritize unit
              cand_unit <- cand[grepl(unit, cand, ignore.case = TRUE)]
              
              # exactly one column --> keep
              if (length(cand_unit) == 1) {
                best <- cand_unit
                
              # no unit column --> shortest
              } else if (length(cand_unit) == 0) {
                best <- cand[which.min(nchar(cand))]
              
              # more than one --> shortest
              } else {
                best <- cand_unit[which.min(nchar(cand_unit))]
              }

            } else {
              # Match all other columns with shortest match
              best <- matched[which.min(nchar(matched))]
            }

            # Apply renaming
            names(df)[names(df) == best] <- clean
          }
        }

        # Keep only relevant columns
        keep_cols <- c("id", clean_names)
        df <- df %>% select(any_of(keep_cols))

        # Convert types
        for (col in names(column_types)) {
          if (col %in% names(df)) {
            df[[col]] <- column_types[[col]](df[[col]])
          }
        }

        df
      }
  }) %>%
  bind_rows()

# Make ID numeric
combined_data$id <- as.numeric(combined_data$id)

# Save final object with parameter name
assign(paste0("combined_", parameter), combined_data)

# Print skipped dataset count
message(skipped, " datasets were skipped due to ambiguous or missing unit matches.")

2.2 Adding relevant metadata

The way in which the parameter was measured is another important aspect that should be taken into account, especially with regard to possible troubleshooting. This information is stored in the metadata. Therefore, we filter the corresponding metadata table by device and add the column to the combined table.

metadata_file <- list.files(
  path = "..",
  pattern = paste0(".*", parameter, ".*", id, ".*metadata.*\\.txt$"),
  ignore.case = TRUE,
  full.names = TRUE
)[1]

metadata <- read.delim(metadata_file) %>%
  select(ID, device) %>%
  rename(id = ID)

# Add (join) the devices to the big table
assign(
  paste0("combined_", parameter),
  get(paste0("combined_", parameter)) %>% left_join(metadata, by = "id") 
)

3 Prepare the data for visualization

When preparing the data for visualisation, spatial and statistical filtering must be performed. This involves sorting out a large amount of data; the exact amount is listed in this step.

3.1 Spatial filtering

Since the bounding box in pangaeapy is based on the coordinates specified in the metadata, the downloaded data sets may contain data points that lie outside of it. Therefore, an additional geographical filter step must be inserted here. First, the bounding box is set again and all data points outside it are removed. Then a polygon of the Baltic Sea is used to filter out the remaining points that lie within the bounding box but outside the Baltic Sea (e.g. northwest of Norway).

Note: You need to download the Polygon for the Worlds Seas here and unzip the files. The baltic_names filter is hard-coded, so for different spatial settings please take that into consideration.

3.2 Statistical filtering

It is necessary to filter out impossible values in order to obtain representative data. Errors can occur when measuring with devices or entering data into tables. However, because this workflow is intended to function for all possible different parameters, it is not practical to set hard-coded limits. A robust method for environmental data is the median absolute deviation (MAD), which is not greatly affected by outliers.

The filter removes items whose values are: |x−median(x)|>k∗MAD(x)|x−median(x)|>k∗MAD(x).
kk is the threshold and usually between 5 (conservative) and 3 (strict).

baltic_simple <- {
  # Load the world seas polygon
  world_seas <- st_read("../World_Seas_IHO_v3/World_Seas_IHO_v3/World_Seas_IHO_v3.shp")

  # Filter for relevant regions
  baltic_names <- c(
    "Baltic Sea",
    "Gulf of Bothnia",
    "Gulf of Finland",
    "Gulf of Riga",
    "Kattegat"
  )

  world_seas %>%
    filter(NAME %in% baltic_names) %>%
    st_make_valid() %>%
    st_union() %>% # merge the sub-polygons to one
    st_sf(geometry = ., crs = st_crs(world_seas)) %>%
    st_simplify(dTolerance = 1500) # simplifying polygon complexity 
}

5. Results and Discussion

PANGAEA presents itself as a FAIR-aligned repository, but several practical issues limit how findable, interoperable, and reusable the data truly are. Date fields are inconsistent, appearing either as pure dates or as date–time combinations, which complicates automated parsing and merging. Many datasets contain multiple columns for the same parameter, often with different units or naming conventions, making automated selection unreliable. The raw dataframes are large and cluttered, and although they become manageable after trimming, the initial heterogeneity slows down processing. Some datasets contain physically impossible values even after statistical filtering, indicating gaps in quality control.

Across all steps, it is difficult to avoid hard‑coding because naming conventions, units, and metadata structures vary widely between datasets. These issues collectively undermine the FAIR principles, especially interoperability and reusability, by shifting the burden of harmonization from the repository to the user. The metadata is curated by PANGAEA, but the datasets themselves are not, meaning that scientists cause inconsistencies that affect the overall data quality.
